## 테스트 계획 수립

### 자체 인터뷰 (Q&A 형식)

#### Q1: 왜 이 테스트 계획서를 작성하나요?
**A**: 이미 개발된 시스템의 안정성을 최종 확인하고, 프로덕션 배포 전 모든 기능이 정상 작동하는지 검증하기 위함입니다. 특히 결제, 영상, 학습 등 핵심 기능의 사용자 경험이 완벽한지 확인해야 합니다.

#### Q2: 누가 이 테스트를 수행하나요?
**A**: 
1. **개발자**: API 테스트, 데이터베이스 검증, 로그 분석
2. **운영자 (박종석 대표)**: 실제 관리자 화면에서 강좌 등록, 회원 관리 테스트
3. **베타 테스터**: 실제 사용자 관점에서 회원가입부터 수강 완료까지 테스트
4. **AI 어시스턴트**: 자동화 스크립트 실행 및 결과 분석

#### Q3: 어떤 순서로 테스트하나요?
**A**: 
1. **인증 시스템** (30분): 회원가입/로그인 → 모든 기능의 전제 조건
2. **강좌 시스템** (30분): 강좌 조회/상세 → 수강 신청의 전제 조건
3. **수강 시스템** (1시간): 수강 신청 → 내 강의실 → 학습 시작
4. **영상 시스템** (1시간): 영상 재생 → 소리 확인 → 진도 저장
5. **결제 시스템** (1시간): 결제 생성 → 승인 → 환불 테스트
6. **수료증 시스템** (30분): 수료 조건 → 발급 → 조회
7. **관리자 시스템** (1시간): 대시보드 → 강좌/회원/결제 관리
8. **통합 시나리오** (1시간): 전체 프로세스 End-to-End 테스트

#### Q4: 결제 시스템은 실제 돈이 빠져나가나요?
**A**: 아니요! 현재 **Toss Payments 테스트 환경**을 사용하고 있어 실제 청구는 발생하지 않습니다. 테스트 카드 번호로 안전하게 테스트할 수 있습니다.

#### Q5: 영상 시스템은 어떻게 테스트하나요?
**A**: 현재 영상 시스템은 **구조만 준비**된 상태입니다. 실제 영상 파일이 업로드되지 않았으므로:
- **현재 테스트 가능**: video_url 필드 확인, 차시 목록 조회
- **테스트 불가**: 실제 영상 재생, 소리 확인
- **권장**: 테스트용 영상 1개를 YouTube Private로 업로드하여 video_url에 입력 후 테스트

#### Q6: 강의 소감 작성은 어디서 하나요?
**A**: 강의 소감(후기) 시스템은 **미구현** 상태입니다. 현재는:
- 수강 신청 → 학습 → 진도율 추적 → 수료증 발급까지만 구현
- 후기 시스템은 추후 추가 개발 필요 (우선순위 낮음)

#### Q7: 5-7시간 안에 모든 테스트를 완료할 수 있나요?
**A**: 네, 가능합니다! 이 계획서는 **효율적인 테스트 순서**를 제시하며:
- **자동화 테스트** (2시간): API 엔드포인트 자동 검증
- **수동 테스트** (3-4시간): 실제 사용자 시나리오
- **문서화** (1시간): 테스트 결과 정리 및 이슈 리포트

## 테스트 시나리오

### 시나리오 1: 신규 회원가입 → 무료 강좌 수강 (20분)

**목표**: 신규 사용자가 회원가입 후 무료 강좌를 수강하는 전체 프로세스 검증

**순서**:
1. 홈페이지 접속
2. [회원가입] 클릭
3. **Google로 계속하기** 버튼 클릭
4. Google 인증 완료
5. 메인 페이지 리다이렉트 확인
6. "무료 체험 과정" 찾기
7. [수강 신청] 클릭
8. "내 강의실" 메뉴 클릭
9. 수강 중인 과정 확인
10. 차시 목록 확인

**예상 결과**:
- ✅ Google 로그인 자동 완료
- ✅ 무료 강좌 즉시 수강 생성
- ✅ 내 강의실에 과정 표시
- ✅ 진도율 0% 표시

**체크포인트**:
```bash
# API 확인
GET /api/enrollments/my
Authorization: Bearer {token}

# 응답 예시
{
  "success": true,
  "data": [
    {
      "id": 1,
      "course_title": "무료 체험 과정",
      "progress_rate": 0,
      "status": "active"
    }
  ]
}
```

### 시나리오 2: 유료 강좌 결제 → 수강 (30분)

**목표**: 유료 강좌 결제부터 학습까지 전체 프로세스 검증

**순서**:
1. 로그인 완료 상태
2. "자기주도학습 지도사 과정" 클릭
3. 강좌 상세 페이지 확인
   - 가격: 300,000원 → 250,000원
   - 수강 기간: 60일
   - 총 20차시
4. [수강 신청] 버튼 클릭
5. 결제 페이지로 이동
6. Toss Payments 위젯 로드 확인
7. 테스트 카드로 결제
   - 카드번호: `4000-0000-0000-0001`
   - 유효기간: 2030년 12월
   - CVC: 123
8. 결제 승인 대기
9. 결제 완료 메시지 확인
10. "내 강의실"로 자동 이동
11. 새로 수강 신청한 과정 확인

**예상 결과**:
- ✅ 결제 위젯 정상 표시
- ✅ 테스트 결제 승인 성공
- ✅ 수강 신청 자동 생성
- ✅ 수강 기간 60일 설정
- ✅ 진도율 0% 표시

**체크포인트**:
```bash
# 결제 생성 API
POST /api/payments-v2
{
  "courseId": 5,
  "amount": 250000
}

# 결제 승인 API
POST /api/payments-v2/confirm
{
```

### 시나리오 6: 환불 처리 (관리자) (15분)

**목표**: 관리자가 환불 요청을 처리하는 프로세스 검증

4. 환불 요청 건 선택

6. [환불] 버튼 클릭
7. 환불 금액 자동 계산 확인
   - 진도 50% 미만 → 50% 환불
   - 125,000원 환불
8. [환불 승인] 클릭
9. Toss API 호출 확인
10. 환불 완료 메시지
11. 결제 상태 "refunded"로 변경 확인

**예상 결과**:
- ✅ 진도율 기반 환불 금액 자동 계산
- ✅ Toss API 환불 호출 성공
- ✅ 결제 상태 업데이트
- ✅ 수강 상태는 유지 (환불 후에도 학습 가능)

**환불 규정**:
```typescript
if (progress_rate === 0 && days_since_enrollment <= 7) {
  refund_amount = 100% // 전액 환불
} else if (progress_rate < 50) {
  refund_amount = 50% // 50% 환불
} else {
  refund_amount = 0% // 환불 불가
}
```

### 이슈 5: 진도율 계산 오류

**증상**:
- 진도율이 100%를 초과함
- 진도율이 음수로 표시됨

**원인**:
```typescript
// 잘못된 계산
progress_rate = (completed_lessons / total_lessons)
```

**해결 방안**:
```typescript
// 올바른 계산
progress_rate = Math.min(100, Math.max(0, 
  Math.round((completed_lessons / total_lessons) * 100)
))
```
